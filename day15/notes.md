Day 15 – Kernel Execution Attribution During Latency Spikes

Introduction
------------
Previous experiments established that the recurring ~200–250 ms latency spikes observed in the Collapse Lab environment
cannot be fully explained by simple resource contention or storage persistence alone.

Day 11 demonstrated that persistence boundaries play an important role in the appearance of the spikes
 by showing that filesystem operations involving durable storage can trigger the phenomenon, 
 while operations executed entirely in memory produce fewer extreme events.

Day 12 further showed that CPU scheduling placement does not eliminate the spikes. 
Even when probe execution was isolated from competing workloads, the characteristic latency class remained visible.

Day 13 introduced additional filesystem configurations and revealed that the spike behavior differs depending on the persistence environment. 
In particular, loop-mounted ext4 configurations significantly reduced the frequency of spikes compared to operations on the root filesystem.

Day 14 added block-device instrumentation to observe storage stack behavior during spike windows.
These experiments showed that block device service activity often coincides with spike events when probes operate on ext4 backed by the microSD device. 
However, when the probe path was moved to tmpfs, which bypasses block-device persistence entirely, the spikes did not disappear.
Furthermore, elevating the probe to real-time scheduling priority did not eliminate the spike class. 
Telemetry also confirmed that CPU pressure, IO pressure, and memory pressure remained negligible during spike windows.
Taken together, these results suggest that the recurring latency spikes are not caused by simple userspace scheduling delay, 
general system resource exhaustion, or block-device persistence alone.
Instead, the observations increasingly point toward a mechanism that originates within kernel execution paths themselves.

The goal of Day 15 is therefore to introduce visibility into kernel-level scheduling activity during spike intervals. Specifically, 
this stage of the investigation focuses on capturing scheduler events that occur when heartbeat latency spikes appear.
By correlating spike timestamps with scheduler trace events such as sched_switch, sched_wakeup, and sched_wakeup_new, 
the experiment aims to characterize how tasks are scheduled during spike windows and whether unusual wakeup or context-switch patterns 
are observable at the time the spikes occur.
Rather than introducing new workloads, 
the experiment maintains the same Collapse Lab framework used in previous days. 
The primary change is the addition of kernel scheduler tracing instrumentation designed to capture system scheduling activity during the spike windows.
This step represents a transition in the investigation from subsystem isolation toward direct observation of scheduler behavior during the latency spikes.

Hypothesis
-----------
The persistent ~200–250 ms latency spike class is generated by a kernel-level scheduling
 or synchronization mechanism that remains active even when block-device persistence is removed and probe scheduling priority is elevated.
Possible kernel-level contributors include:
	- delayed scheduler wakeup or delayed wakeup completion within the kernel scheduler
	- interrupt-related execution delays
	- filesystem or writeback related kernel serialization
	- shared kernel synchronization points that temporarily block execution progress

Day15 focuses specifically on the scheduler component of this hypothesis.
By tracing scheduler events (sched_switch, sched_wakeup, 
sched_wakeup_new) during spike windows, 
the experiment examines whether heartbeat latency spikes temporally coincide with identifiable scheduler activity patterns.

Under this hypothesis, spike timestamps should align with observable kernel scheduler events or scheduling activity patterns, 
even when userspace scheduling priority and storage persistence effects are minimized.

Methods
-------
Day15 maintains the same workload structure used in Day14 in order to avoid introducing new behavioral variables.

The experiment continues to use the known reproducer:
	- PROBE_B
	- B_variant=fsync
	- OFF=3
	- N_CYCLES=3

The same cycle structure is preserved:
	- baseline
	- intervention
	- recovery R1
	- recovery R2
	- post-baseline

This ensures that any new observations can be attributed to added kernel tracing rather than changes in the workload itself.

Control configuration
----------------------
Because Day14 showed that the spike class persists even when the probe path is moved to tmpfs, 
Day15 uses the tmpfs probe configuration as the primary tracing condition.
This choice reduces storage-path amplification while preserving the core spike class, 
making it easier to observe kernel scheduling activity associated with the remaining latency mechanism.

Primary Day15 tracing configuration:
- PROBE_ROOT=/dev/shm/day15_probes
- PROBE_B_VARIANT=fsync
- normal scheduling priority
- full Day14 telemetry retained

A secondary ext4 tracing run may be performed later only if needed for comparison.

Signals collected
------------------
Day15 extends the Day14 instrumentation by collecting five synchronized signal classes.

1. Heartbeat timing
File: heartbeat.log

This remains the reference signal for spike detection.

2. Probe timing

File: probes.log

Used to identify probe boundaries and align spikes relative to PROBE_B_START and PROBE_B_END.

3. Telemetry

File: telemetry.log

Retained from Day14 to monitor:

PSI

VM counters

dirty/writeback state

disk activity counters

4. Disk activity

File: diskstats.log

Retained from Day14 for continuity and comparison with earlier storage-correlated spikes.

5. Kernel trace

File: trace.log

This is the new Day15 signal.

Kernel tracing captures scheduler events occurring during spike windows.

Kernel tracing configuration
------------------------------
The objective of Day15 is targeted observation of scheduler activity during spike intervals rather than broad kernel profiling.
Tracing therefore focuses on a small set of scheduler tracepoints.

	* Scheduler events collected:
	* sched_switch
	* sched_wakeup
	* sched_wakeup_new

These events allow observation of:
	- task wakeups
	- context switches
	- newly created runnable tasks
	- CPU assignment of scheduled tasks

Trace collection window
------------------------
Kernel tracing is enabled only during the R2 phase.

Rationale:

	* R2 is the phase where the spike phenomenon is actively probed
	* limiting tracing reduces trace size
	* limiting tracing reduces the risk that tracing affects earlier phases
	* Tracing starts at C?_RECOVERY_R2_START and stops at C?_RECOVERY_R2_END.

This keeps the traced window aligned with the existing telemetry and disk observation windows.

Trace mechanism
----------------
Tracing uses ftrace via tracefs, which provides low overhead kernel tracepoints.
Trace configuration per cycle:

	- clear existing trace buffer
	- enable selected scheduler events
	- enable tracing at R2 start
	- disable tracing at R2 end
	- save trace buffer to the cycle directory

Example trace output files:

cycle_1/trace.log
cycle_2/trace.log
cycle_3/trace.log

Data alignment method
-----------------------
After the run completes, analysis correlates:

* heartbeat spike timestamps
* nearest probe events
* nearest telemetry samples
* nearest diskstats samples
* nearest scheduler trace events

The purpose of this alignment is to identify scheduler activity occurring near the observed spike timestamps.

Expected limitations
---------------------
Several limitations are acknowledged:

- tracing may introduce small overhead
- scheduler tracepoints provide only partial visibility into kernel behavior
- some kernel paths may not be visible through scheduler events alone
- timestamp alignment may require tolerance windows rather than exact matches

For this reason, Day15 is treated as an initial kernel-observability step rather than a complete attribution of kernel execution paths.

Results
--------
Spike occurrence during the Day15 run

The Day15 experiment completed three full cycles using the tmpfs probe configuration described in the Methods section.

Heartbeat latency measurements recorded a small number of spikes exceeding 200 ms during the run.
Across the three cycles, four spikes above the 200 ms threshold were observed.

Distribution of spikes by phase:

Phase		Spike count
Baseline		0
Intervention	3
Recovery R1		0
Recovery R2		0
Post-baseline	1

Each intervention phase contained one spike exceeding 200 ms.
An additional spike occurred during the post-baseline phase.

The majority of heartbeat intervals remained within the normal low-latency range outside of these events.

Trace capture
--------------
Kernel tracing was enabled during the R2 phase of each cycle.
Trace logs were successfully captured for all three cycles:

cycle_1/trace.log
cycle_2/trace.log
cycle_3/trace.log

The traces were collected using ftrace via tracefs with the scheduler tracepoints:
	sched_switch
	sched_wakeup
	sched_wakeup_new

Example trace header from the captured logs:
entries-in-buffer/entries-written: 99004/1918912
CPUs: 4

Trace entries contain the following fields:

TASK-PID  CPU  FLAGS  TIMESTAMP  EVENT

Example entry:

bash-957716  [002] d..2.  8724.784252: sched_switch

The traces show scheduler activity across all four CPU cores.

Scheduler activity during spike windows
-----------------------------------------
Spike timestamps obtained from the heartbeat log were aligned with the nearest scheduler trace events recorded during the R2 trace window.
Scheduler activity in the vicinity of spike timestamps consisted primarily of repeated sequences of:
	sched_wakeup
	sched_switch

These sequences typically involved transitions between idle tasks (swapper/*) and runnable tasks.

Example pattern observed in the trace:

swapper -> rcu_preempt
rcu_preempt -> swapper
swapper -> bash
bash -> swapper

Scheduler activity was recorded on multiple CPUs during these intervals.

Tasks active during spike intervals
-------------------------------------
Tasks observed in the scheduler traces near spike timestamps included both kernel threads and short-lived user-space processes.

Kernel tasks observed:
rcu_preempt
kworker/*
ksoftirqd/*
swapper/*

User-space tasks observed:
bash
sleep
date
awk
grep

Several of the user-space tasks appear as newly created processes (sched_wakeup_new) followed shortly by task termination states (prev_state=Z).

Example sequence:
sched_wakeup_new: comm=bash pid=957715
sched_switch: prev_comm=grep prev_state=Z

These events indicate short-lived processes executing briefly before exiting.

CPU distribution of scheduler events
--------------------------------------
Scheduler activity during the traced windows occurred across all four CPU cores.

Examples observed in the trace:

CPU		Example tasks
CPU0	bash, awk, rcu_preempt
CPU1	bash, sleep
CPU2	bash, kworker, ksoftirqd
CPU3	rcu_preempt, bash, sleep

Context switches frequently involved transitions between idle tasks and runnable tasks on the same CPU.

Example trace pattern:

swapper/2   -> bash
bash 	    -> swapper/2
swapper/3   -> rcu_preempt
rcu_preempt -> swapper/3

Event density near spike timestamps
------------------------------------
Within the time windows surrounding spike timestamps, the trace logs show dense sequences of scheduler events.

Example sequence from a spike window:

8724.789545: sched_switch
8724.789578: sched_switch
8724.789583: sched_wakeup
8724.789585: sched_switch

These events occur at sub-millisecond intervals and reflect rapid context switching activity among runnable tasks.

Conclusions
------------
Day15 introduced kernel scheduler tracing into the Collapse Lab investigation in order to observe kernel activity occurring during latency spike intervals.
The experiment preserved the workload configuration used in Day14 
while adding targeted scheduler instrumentation using ftrace tracepoints (sched_switch, sched_wakeup, 
and sched_wakeup_new). 
Tracing was limited to the R2 phase of each cycle in order to capture kernel activity 
during the portion of the cycle where the spike phenomenon is actively probed while minimizing tracing overhead.
Heartbeat measurements during the run recorded four latency spikes exceeding 200 ms across the three experimental cycles. 
Three spikes occurred during intervention phases and one spike occurred during the post-baseline phase. 
The spike class therefore continues to appear under the tmpfs probe configuration used in Day15, 
confirming earlier observations that the phenomenon persists even when block-device persistence is removed from the probe path.
Kernel scheduler traces were successfully captured for all cycles. 
Examination of the trace logs shows continuous scheduler activity during the traced intervals, 
including frequent sched_switch and sched_wakeup events across multiple CPU cores. 
Tasks observed in the trace include both kernel threads (rcu_preempt, kworker, ksoftirqd, 
and idle tasks) and short-lived user-space processes created by the probe framework.
Alignment of spike timestamps with the scheduler trace windows shows that spikes occur while normal kernel scheduling activity is ongoing. 
The traces reveal repeated sequences of task wakeups and context switches during the intervals surrounding spike timestamps, 
indicating that the CPU is actively executing kernel and user tasks rather than remaining idle during these periods.
However, the scheduler tracepoints used in this experiment provide visibility only into task wakeups and context switches 
and do not expose internal execution paths within kernel subsystems. As a result, 
although scheduler activity is observable during spike windows, 
the traces do not by themselves identify the specific kernel mechanism responsible for the latency spikes.

The Day15 experiment therefore establishes two important results. 
First, the characteristic ~200–250 ms latency spike class persists under tmpfs conditions and 
continues to appear during the intervention phases of the Collapse Lab reproducer. 
Second, kernel scheduler tracing can be successfully aligned with spike timestamps and provides direct visibility into 
system scheduling behavior during these events.
These observations demonstrate that the Collapse Lab framework can now correlate heartbeat latency spikes with kernel-level execution activity. 
Further investigation will require extending kernel instrumentation to additional subsystems in order to determine whether filesystem synchronization, 
interrupt handling, kernel synchronization primitives, or other execution paths contribute to the observed latency behavior.
